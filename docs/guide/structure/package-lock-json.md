# 📄 package-lock.json

Файл `package-lock.json` автоматически генерируется npm при установке зависимостей и хранит «снимок» (lockfile) всего
дерева зависимостей проекта.

## Основные задачи

- **Фиксирует версии** — гарантирует, что все разработчики и CI будут использовать абсолютно одинаковые версии пакетов, независимо от того, вышли ли новые минорные или патч-версии.
- **Ускоряет установку** — npm использует lock-файл как кэш: он точно знает, какие версии и откуда скачивать.
- **Повышает безопасность** — инструменты аудита (`npm audit`) проверяют именно lock-файл, чтобы анализировать реальные используемые версии, а не только «диапазоны» из `package.json`.
- **Делает сборку воспроизводимой** — одна и та же версия проекта в любой среде даст идентичный набор node_modules.

## Содержимое

- **name, version** — имя и версия корневого проекта.
- **lockfileVersion** — версия формата lock-файла (меняется в зависимости от версии npm).
- **dependencies** — полный список всех зависимостей, включая транзитивные (т.е. те, что подтянулись через другие пакеты).
- Для каждой зависимости фиксируется:
  - точная **версия** (`version`),
  - источник загрузки (`resolved`, например ссылка на npm registry),
  - **хэш целостности** (`integrity`, для проверки на подмену),
  - дополнительные флаги (`dev`, `optional`, `peer`).

## Почему файл хранится в Git

- Чтобы разработчики и CI использовали **одинаковую среду**.
- Чтобы исключить «дрейф» зависимостей при повторной установке.
- Чтобы деплой на сервер был предсказуемым и не зависел от обновлений в npm.
- Чтобы ускорить установку — lock-файл позволяет npm пропускать многие шаги резолвинга.

## Особенности

- Файл **довольно большой** — это нормально для современного фронтенда, так как каждая основная библиотека тянет десятки транзитивных зависимостей.
- Так как он **всегда в Git**, любые изменения зависимостей (даже транзитивных) фиксируются в истории коммитов, и можно отследить, что и зачем изменилось.
